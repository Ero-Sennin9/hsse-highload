## Часть 2: Доступность

### 1. Целевая доступность системы

#### 1.1. Сводная таблица по контурам

| Контур | Целевая доступность | SLI | Обоснование |
|--------|---------------------|---------------------|-------------|
| **Каталог и поиск** (read path) | **99.99%** в год (~≤ 52,6 мин простоя) | Доля HTTP 2xx на поиске и карточках (НФТ-003) | Поиск — главная страница; недоступность блокирует большинство сессий при 20 млн MAU |
| **Транзакции и оплаты** (write path биллинга) | **99.95%** для API приёма платежа; **0% потерь** данных (отдельное SLO) | Потерянные записи о транзакциях = 0 (НФТ-002) | Потеря платежа = прямые убытки и юридические риски; доступность API и сохранность данных — разные метрики |
| **Создание/редактирование объявлений** | **99.9%** | Косвенно: write path, p99 write в PoC < 1 с | Сбой write не «убивает» весь трафик (read-heavy ~80%), но блокирует монетизацию |
| **Модерация (async)** | **99.5%** (фоновый контур) | p95 решения < 5 с (НФТ-004) | Очередь модерации замораживает публикации и доход от продвижения, но не ломает просмотр каталога |
| **Медиа (CDN + S3)** | **99.9%** | p99 миниатюры < 150 мс (НФТ-005) | При сбое фото текст карточки остаётся; риск Medium |
| **Email-уведомления** | **99.0%** | p95 доставки события < 5 мин (НФТ-006) | Low business risk; задержка не ломает основной флоу |


#### 1.2. Бизнес-обоснование

| Фактор | Следствие для доступности |
|--------|---------------------------|
| **Поиск = главная страница** | Задержки и простой read path → «моментальный отток» (НФТ-001, High) |
| **Каталог недоступен** | «Блокирует абсолютное большинство пользовательских сессий» (НФТ-003, High) |
| **Потеря платежей** | Прямые финансовые убытки + юридические риски; в архитектуре — outbox + idempotency (ADR-003, НФТ-002) |
| **Очередь модерации** | Замораживает публикации и доход от продвижения (НФТ-004), но не отменяет просмотр уже опубликованного |
| **Профиль трафика** | Read-heavy (~80% GET) — инвестиции в HA read path (ES, Redis, CDN, реплики) окупаются сильнее, чем в write |
| **Масштаб** | 20 млн MAU, пиковый read; каждый час простоя каталога при полном отказе — порядка **миллионов** незавершённых сессий просмотра (оценка: 20M / 24 ≈ 830K MAU-часов активности в сутки, доля онлайн в пике выше) |

**Оценка «сколько денег теряется при простое»**:

- **1 час полного простоя каталога** в пик: недополученные просмотры → падение показов продвигаемых объявлений → потеря комиссии за продвижение и рекламного инвентаря; вторично — отток MAU (долгий эффект).
- **1 час недоступности только write (создание/оплата)** при живом read: пользователи продолжают искать, но **новые** объявления и платежи не проходят — прямой удар по выручке дня (~15% write + ~5% promotion в профиле PoC).
- Для обоснования **99.99%** на read: при ARPU/рекламной выручке порядка сотен млн ₽/год каждые **лишние 9 минут** простоя сверх бюджета SLO (сравнение с 99.95%: ~4,4 ч/год vs ~52,6 мин/год) — уже заметная статья; плюс репутационный риск для маркетплейса-«главной страницы».

**Ожидания пользователей:** поиск и карточка должны открываться «всегда»; оплата должна либо пройти, либо явно отклониться без двойного списания.

### 2. RPO и RTO по критичным компонентам

| Компонент | Роль | **RPO** | **RTO** | Комментарий |
|-----------|------|---------|---------|-------------|
| **PostgreSQL** | Source of truth: объявления, платежи, outbox, подписки | **0** (синхронная реплика) или **< 1 мин** (async + WAL) | **15–30 мин** (авто-failover Patroni/k8s operator) | НФТ-002: 0% потерь; outbox в той же БД (ADR-003). Потеря WAL недопустима для платежей |
| **Kafka** | События домена, outbox publisher, модерация, индексация | **0** при `acks=all` и min.insync.replicas=2; практический потолок — непубликованные сообщения в outbox до replay | **30–60 мин** (подъём кластера / переключение брокеров) | Retention ≥ 7 дней для replay; DLQ для poison messages (ADR-003) |
| **Redis** | Кэш карточек, rate limit counters | **N/A** (кэш) / **< 5 мин** если хранит лимиты | **5–15 мин** | Промах кэша → нагрузка на PG/ES; не источник истины |
| **Elasticsearch** | Read-модель поиска (CQRS, ADR-002) | **2–3 мин** / при полной потере индекса — **часы** (полный reindex из PG/Kafka) | **30–60 мин** (failover реплик) / **4–24 ч** (полное пересоздание индекса) | Данные восстанавливаются из write-модели; idempotent indexer |
| **S3/MinIO + CDN** | Медиа | **0** (версионирование bucket / cross-region replication для DR) | **15–60 мин** (переключение CDN origin) | НФТ-005; оригиналы после пережатия удаляются — важны thumbnail-ключи |
| **Stateless API** (ads, search, billing, gateway) | HTTP | **N/A** | **5–10 мин** (rolling deploy, новые pods) | Healthcheck + отказ readiness при недоступной БД (`GET /health` + Ping PG) |

### 3. Стратегия резервирования

#### 3.1. По компонентам

| Компонент | Стратегия | Геораспределённость |
|-----------|-----------|---------------------|
| **PostgreSQL** | **Active/standby** (primary + 1–2 sync/async replica), автоматический failover; read replica для тяжёлых отчётов | **Одна региональная зона** на старте (MVP/PoC); **cross-AZ** внутри региона обязательно для 99.99%. **Active/active multi-region** для PG — фаза 2 (конфликты записи, стоимость) |
| **Kafka** | **Active/active** кластер (≥ 3 брокера, RF=3, rack-aware) | Cross-AZ в регионе; geo-replication (MirrorMaker 2) — только при требовании DR на уровне региона |
| **Redis** | **Active/active** (Redis Cluster или Sentinel, 3+ узла) | Cross-AZ; без geo на MVP (кэш пересобирается) |
| **Elasticsearch** | **Active/active** (кластер с репликами шардов, min 3 data-node) | Cross-AZ; snapshot в object storage — для DR индекса |
| **API Gateway + сервисы** | **Active/active** (N≥2 инстанса за LB, HPA) | Cross-AZ; stateless |
| **S3 + CDN** | Multi-AZ bucket; CDN — edge PoPs | Geo CDN по умолчанию; bucket replication — при DR второго региона |

#### 3.2. Нужна ли геораспределённость

| Этап | Решение |
|------|---------|
| **PoC / учебный стенд** | Нет: один хост, `postgres:16-alpine` без реплик (`docker-compose.yml`) |
| **Production MVP (20 млн MAU, один рынок)** | **Достаточно одного региона + 3 AZ**: закрывает отказ дата-центра/AZ и поддерживает 99.99% при rolling-обновлениях |
| **Production scale / DR «регион целиком»** | Второй регион: read-only реплики PG, Kafka MM2, ES snapshot restore, CDN failover — если SLA контрактует DR при падении региона |

### 4. Maintenance window

#### 4.1. Есть ли окно

| Среда | Maintenance window |
|-------|-------------------|
| **Production (целевое)** | **Формального длительного окна нет.** 99.99% SLO на read не оставляет ~еженедельных часов простоя. Обслуживание — через **rolling updates**, **blue/green** или **canary** для stateless-сервисов |
| **Исключения (инфраструктура с состоянием)** | Короткие окна только для операций, несовместимых с online failover (major PG upgrade, ES mapping migration): **≤ 15 мин**, согласованные с низкой нагрузкой |

#### 4.2. Рекомендуемое окно

| Параметр | Значение |
|----------|----------|
| **День** | Вторник или среда (избегать пятницы–воскресенья и понедельника утра) |
| **Время (MSK)** | **03:00–05:00** — минимум трафика для маркетплейса |
| **Частота** | Не чаще **1/месяц** для рискованных изменений БД; **еженедельно** — только rolling deploy приложений без простоя |
| **Допустимый простой в окне** | **0 мин** для read path (переключение на replica); **≤ 15 мин** для редких PG/ES maintenance при отсутствии автоматического failover |

### Что дороже всего и почему

**Kafka — 35% инфраструктурных затрат** при baseline. На текущей нагрузке Kafka утилизирована слабо (только outbox промоушенов из ADR-003), но оплачивается 6 хостов (3 брокера + 3 ZooKeeper) полностью. Это наиболее дорогая статья именно из-за минимального количества хостов для кворума — нельзя опуститься ниже трёх брокеров.

**PostgreSQL + Elasticsearch — 40% совместно**. Оба компонента требуют SSD и репликации. При росте ×5 PostgreSQL дорожает нелинейно: вертикальный апгрейд класса хоста значительно дороже горизонтального масштабирования через read-реплику. Именно поэтому ADR-002 (read-path на Elasticsearch) снижает давление на PostgreSQL при росте read-нагрузки.

**Как снизить затраты:**
- CVoS на 1 год — скидка ~22% на Compute и PostgreSQL
- Прерываемые ВМ для ads/billing — экономия ~60% на Compute при наличии retry в ALB
- До нагрузки ×3 можно не держать Kafka, используя polling-модель outbox (экономия ~21 000 ₽/мес)
- Object Storage Cold для медиафайлов старше 30 дней — стоимость хранения вдвое ниже
