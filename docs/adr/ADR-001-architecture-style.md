# ADR-001: Hybrid SBA + EDA + CQRS architecture

## Status

Accepted

## Context

Система маркетплейса объявлений должна одновременно:
- держать быстрый поиск (`p99 < 200 ms`);
- обеспечивать надежный платежный контур (0% потерь платежей);
- масштабироваться при росте MAU;
- поддерживать асинхронную модерацию и уведомления.

Монолитная архитектура усложнит масштабирование поиска и изоляцию платежного контура.

## Alternatives

1. **Layered Monolith**
   - Pros: проще старт и эксплуатация.
   - Cons: слабая изоляция контуров, сложный масштаб под пиковый read/write.

2. **Pure Microservices (fine-grained)**
   - Pros: максимальная независимость команд и сервисов.
   - Cons: слишком высокая operational сложность для текущего этапа.

3. **Hybrid SBA + EDA + CQRS**
   - Pros: баланс между масштабированием, надежностью и стоимостью изменений.
   - Cons: появляется eventual consistency и сложнее observability.

## Decision

Выбираем `Hybrid SBA + EDA + CQRS`:
- SBA для крупных бизнес-компонентов;
- EDA через Kafka для интеграции и фоновых процессов;
- CQRS для поискового контура (PostgreSQL write + Elasticsearch read).

## Consequences

### Positive
- Масштабируемый поиск и независимый read path.
- Изоляция платежного контура и отказов.
- Возможность эволюционного роста без полного переписывания.

### Negative
- Сложнее сопровождение и отладка межсервисных сценариев.
- Нужна дисциплина схем событий, ретраев и идемпотентности.
