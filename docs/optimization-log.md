### Bottleneck (iter-0)

**Bottleneck не выявлен** — система не нагружена.

VM с 2 vCPU справляется с нагрузкой до 600 RPS при утилизации CPU менее 30% и RAM менее 15%. Ни один ресурс не стал узким местом на текущем диапазоне нагрузки. Ступени stress.js до 600 total RPS прошли без единой ошибки.

Возможные причины низкой нагрузки на ресурсы:
- Данные в объявлениях небольшие (только текстовые поля), PostgreSQL легко держит их в shared buffers
- Connection pool Go-сервиса (10 соединений) не исчерпывается
- HDD не задействован активно (БД помещается в RAM)

## Iteration 1 — Поиск потолка (stress до 1500 RPS)

Дата: 2026-05-17. Цель: найти реальный bottleneck, подняв нагрузку выше baseline.

### Что изменили

- `loadtest/stress.js`: расширили ступени до 1500 total RPS (reads 1200 + writes 225 + promotions 75 iter/s).

### Bottleneck

На ступени ~900–1200 total RPS write-путь (`POST /api/v1/ads`) начал отдавать `dial: i/o timeout`.

**Причины:**
1. `ads-service` CPU-лимит `0.50` cores — goroutine-пул исчерпывался под нагрузкой.
2. `db.SetMaxOpenConns(10)` — PostgreSQL connection pool слишком мал при burst write.
3. Rate limiter `rate.Limit(120), burst=40` — срабатывал раньше, чем нужно.

### Результаты iter-1 (stress.js, extended)

| Метрика              | Iter-0 (600 RPS max) | Iter-1 (1500 RPS target) | Delta    |
|----------------------|----------------------|--------------------------|----------|
| http_req_failed      | 0.00%                | **2.83%**                | +2.83%   |
| p(90) общий          | ~27 мс               | ~200+ мс                 | хуже     |
| Ошибки               | нет                  | `dial: i/o timeout`      | деград.  |
| Bottleneck           | нет                  | **CPU ads-svc + conn pool** | найден |

## Iteration 2 — Оптимизация write-path

Дата: 2026-05-17. Цель: устранить bottleneck из iter-1.

### Что изменили

| Файл | Было | Стало | Зачем |
|------|------|-------|-------|
| `docker-compose.yml` | `cpus: "0.50"` | `cpus: "1.00"` | ads-service перестаёт упираться в CPU |
| `adapters/postgres/conn.go` | `MaxOpenConns(10)`, `MaxIdleConns(4)` | `MaxOpenConns(25)`, `MaxIdleConns(10)` | больше параллельных запросов к PG |
| `presentation/http/handlers.go` | `rate.Limit(120), burst=40` | `rate.Limit(500), burst=100` | rate-limiter не режет write раньше времени |

### Результаты iter-2 (stress.js, та же нагрузка 1500 RPS target)

| Метрика              | Iter-1 (до)   | Iter-2 (после) | Delta     |
|----------------------|---------------|----------------|-----------|
| http_req_failed      | **2.83%**     | **0.00%** ✓    | −2.83%    |
| p(90)                | ~200+ мс      | **91.57 мс**   | −55%+     |
| p(95)                | ~400+ мс      | **134.3 мс**   | −66%+     |
| avg latency          | деградирован  | **50.49 мс**   | восстановлен |
| Total requests       | ~400k         | **563 547**    | +41%      |
| Ошибки               | i/o timeout   | **нет**        | устранены |
| NFR Read p99 < 500мс | ✗ (degraded)  | ✓              | выполнен  |
| NFR Write p99 < 1000мс | ✗ (timeout) | ✓              | выполнен  |

**Вывод:** увеличение CPU-лимита и connection pool полностью устранили bottleneck. Система уверенно держит 1200 read RPS + 225 write RPS при 0% ошибок.
